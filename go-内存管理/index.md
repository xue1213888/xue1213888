# go语言内存管理 - 系统内存管理


# 最原始的内存操作

我们可以吧内存看作是一个连续的数组，内存地址理解为数组的下标，我们一开始CPU是直接通过内存地址来访问物理内存的。

慢慢多任务的需求，希望多个任务能同时在系统上运行，就会引发一系列问题。

- 内存访问冲突：多个程序访问同一块内存，数据读写混乱，程序崩溃。
- 内存不够用：每个程序都要为自己划走一块单独的内存空间，哪怕用不了都得按照自己需要的最大需求量来获取。
- 开发成本提高：对于内存的计算很费脑子。

# 虚拟内存

虚拟内存的出现很好的解决上面的问题，用户只能通过虚拟内存来获取数据，系统会将虚拟地址翻译成物理地址。

所有的程序都好像单独的独享一块内存一样进行使用，程序不用太关心物理内存的冲突。

内存不够用的话，虚拟内存本质上是将本地磁盘当成最终存储，而主存作为一个cache。

程序从虚拟内存申请一块很大的空间，但物理内存不会开辟那么大，只会开辟很小的一点。

这样程序在访问内存时，操作系统看访问的虚拟地址能否翻译为物理内存地址，不能的话，再开辟新的内存空间。

本质上就是程序经常访问的数据存在内存里面加速访问，不常用的数据放在了磁盘上。用户也可以感知到存储位置，也可以通过虚拟内存地址去访问数据，将管理权交给了操作系统。

# 虚拟地址的翻译

虚拟内存的实现方式，大多都是通过`页表`来实现的。操作系统将虚拟内存空间分成一页一页来进行管理，每页大小为4k。磁盘和主存之间的置换也是以页为单位。4k是实践出来的通用值，太小了需要频繁置换，太大了又浪费内存。

`虚拟内存->物理内存`的映射关系由叶表进行维护，这就是一个数组，数组的每个元素叫做页表条目，就是目录，简称PTE，PTE由一个有效位和n个地址字段构成，有效位标识了这个虚拟地址是否分配了物理内存。

页表被操作系统放在物理内存的指定位置，CPU上有一个内存管理单元MMU，CPU把虚拟地址给MMU，MMU去物理内存中的页表查询得到实际地址。MMU也不会每次去查，它有自己的一份缓存叫做TLB，是为了加速地址翻译。

## CPU访问过程

1. CPU使用虚拟地址访问数据，将虚拟地址送给了MMU。
2. MMU查询页表或者自己的cache获取PTE。
3. 如果PTE是真实的物理内存地址，流程到此结束。
4. 如果PTE没有关联的物理内存，MMU触发一个缺页异常。
5. 操作系统捕获异常，执行异常处理函数，在物理内存创建一页内存，并更新页表。
6. 缺页处理程序在物理内存确定一个牺牲页，如果牺牲页上有数据，就把数据保存到磁盘。
7. 缺页处理程序更新PTE。
8. 再去执行相关的地址访问。

# 内存命中率

上面如果到第四步就很繁琐了，效率也很差，我们称第三步结束的叫做命中，未结束的就叫做未命中。命中率越高证明这个内存管理程序越好。

如果物理内存不足，就会频繁的从磁盘进行置换数据，命中率很低，性能急剧下降，叫做内存颠簸，swap利用率开始增高，CPU `iowait`占比增高。

## 程序的局部性

局部性是说应用过一次的存储器的位置，在后续很可能呢会经常访问，并且该附近的其他区域也是高频访问区域。

计算机的级级缓存就使用了局部性这个特点，没有用到这个特点的程序，会比其他程序慢。

# CPU Cache

随着技术发展，CPU运算速度越来越快，但是内存访问速度没有什么突破，导致CPU访问主存成了整个计算机运行性能的瓶颈。CPU Cache就是为了解决这个问题，在CPU和主存中间加入了Cache，用来缓存一块内存中的数据，而且还不止一个，现在计算机一般都有三级缓存，其中L1 Cache的访问速度和寄存器差不多。

系统会假定程序都具有局部性，如果某一级缓存未命中，它会将该级存储的数据更新为最近使用的数据。

主存与磁盘是以page为单位通常4k大小进行交换，CPU Cache与主存之间是以Cache line 通常64byte进行交换。

# 程序内存布局

最后看一下程序的内存布局，我们现在知道了每个程序都有自己一套独立的虚拟地址空间。操作系统已经将一整块内存划好了区域，每个区域做不同的事。

1. text段：存储程序的二进制指令，和一些静态资源。
2. data段：存储已经被初始化的全局变量。
3. bss段：存储未被初始化的全局变量。静态分批，在编译期间就确定了大小，不会释放。
4. stack段：栈空间，主要用于函数存储临时变量。自动分配自动释放。
5. heap段：堆空间，动态分配，go语言主要靠GC管理这部分，

起始远比上面五个复杂的多，但是我们从开发程序的角度，只关心stack段和heap段就好了。堆空间虽然golang帮我们管理了，但是代价也很高。
